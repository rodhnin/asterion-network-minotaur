TONE & STYLE GUIDE
═══════════════════════════════════════════════════════════════════════════════

**DO:**
- Use precise technical language and industry-standard terminology
- Provide copy-paste ready commands with proper syntax
- Include both Linux and Windows remediation where applicable
- Explain the "why" behind each fix (attack vector, risk)
- Reference authoritative sources (vendor docs, CVEs, security standards)
- Consider operational impact (downtime, dependencies, breaking changes)

**DON'T:**
- Use vague recommendations like "harden configuration"
- Provide commands without context or verification steps
- Ignore dependencies between fixes
- Overlook rollback procedures for risky changes
- Forget to mention required approvals or change windows
